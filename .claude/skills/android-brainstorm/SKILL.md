# Brainstorm

## 概述

通过结构化提问帮助用户想清楚。不写代码，只产出思考。

**启动时声明:** "我正在使用 android-brainstorm skill。"

## 调用方式

```bash
/android-brainstorm                         # 交互式: 输入主题 → 发散 → 思考文档
/android-brainstorm <主题描述>              # 直接传入主题
/android-brainstorm resume <file>          # 从保存的思考文档恢复，继续发散
```

**零外部依赖:** 不依赖 gstack、Codex、browse、design。仅使用 Claude Code 原生工具。

## 核心原则

1. **先理解再发散。** 不急于给方案，先把问题想清楚。
2. **一个问题问到底。** 不同时问多个问题，逐个深入。
3. **具体性是货币。** 模糊的答案被追问，具体的答案被认可。
4. **挑战假设。** 每个前提都值得质疑。
5. **不发明，只发现。** 帮用户发现自己已经知道的答案。
6. **验证而非猜测。** 发散完用外部信息检验，用独立视角找盲点，用自我检验堵漏洞。
7. **记忆跨会话。** 思考文档可选保存到文件，下次直接恢复。

## Phase 0: 项目上下文感知

在开始提问之前，快速扫描项目上下文，让后续提问更有针对性。

### 步骤 1: 读取项目配置

```bash
# 读取项目约定
[ -f "CLAUDE.md" ] && echo "HAS_CLAUDE_MD" || echo "NO_CLAUDE_MD"
[ -f "TODOS.md" ] && echo "HAS_TODOS" || echo "NO_TODOS"
[ -f "ARCHITECTURE.md" ] && echo "HAS_ARCHITECTURE" || echo "NO_ARCHITECTURE"
```

如果有 `CLAUDE.md`，读取其中的项目约定、技术栈约束、架构说明。
如果有 `TODOS.md`，读取当前待办事项。
如果有 `ARCHITECTURE.md`，读取架构决策。

### 步骤 2: 读取代码结构

```bash
# 快速了解项目类型
ls *.gradle *.gradle.kts 2>/dev/null | head -3        # Android?
ls package.json 2>/dev/null | head -1                        # Node.js?
ls go.mod 2>/dev/null | head -1                             # Go?
ls Cargo.toml 2>/dev/null | head -1                         # Rust?
ls pyproject.toml setup.py 2>/dev/null | head -2             # Python?
```

```bash
# 最近的代码活动
git log --oneline -10 2>/dev/null
```

### 步骤 3: 检查已有思考文档

```bash
# 查找之前保存的思考文档
find . -name "*-thinking*.md" -o -name "*-android-brainstorm*.md" 2>/dev/null | head -10
ls docs/thinking/ 2>/dev/null
```

如果找到相关的已有思考文档，告知用户:
> 找到之前的思考文档: [文件名] ([日期])。要基于它继续发散吗?

## Phase 1: 明确目标

在发散之前，先搞清楚用户到底想达成什么。

### 步骤 1: 接收主题

**交互式:** 使用 AskUserQuestion:
> 你想探讨什么? 可以是功能需求、技术方案、产品方向、架构设计、任何需要想清楚的事情。

**带参数:** 直接使用传入的主题描述。

**resume 模式:** 读取指定的思考文档，恢复上下文后跳到当前状态。

### 步骤 2: 目标锚定

问第一个问题，把模糊的想法变成具体的目标。

**问:** "用一句话描述，完成这件事之后，理想的结果是什么样的?"

追问方向:
- 结果可以被验证吗? (怎么知道做成了?)
- 谁会受益? (不是"用户"，是具体的人)
- 如果只做一件事就能达成目标，那件事是什么?

**判断:**
- 用户能用一句话说清楚 → 进入步骤 3
- 用户说了三句以上 → 帮用户提炼，确认后再进入

### 步骤 3: 边界探测

搞清楚目标的边界。什么东西在范围内，什么东西不在。

问以下问题 (根据主题选择最相关的 1-2 个):

**范围边界:**
- "这个东西做到什么程度就够了? 什么程度是过度?"
- "如果有时间只做 20% 就能获得 80% 的价值，那 20% 是什么?"

**时间边界:**
- "这个事情有截止时间吗? 如果没有，你觉得多久是合理的?"
- "如果明天就要交出一个版本，你会砍掉什么?"

**质量边界:**
- "做到什么程度你会说'够好了'? 完美主义者会说'永远不够'，但你需要一个标准。"

**反边界 (什么坚决不做):**
- "有什么是你明确不想做的? 即使它能带来价值。"
- "有什么方案是你已经排除的? 为什么?"

**输出:**
```
目标: <一句话>
范围: <在范围内 / 在范围外>
边界: <做到 X 就够了>
反边界: <坚决不做 Y>
```

使用 AskUserQuestion 确认:
> 这是你想达成的目标吗?
> - A) 是，继续
> - B) 不完全对，让我补充

## Phase 2: 发散思维

目标明确后，开始发散。根据主题的性质选择模式。

### 模式判断

根据 Phase 0 和 Phase 1 收集的信息，判断使用哪种模式:

| 模式 | 适用场景 | 特征 |
|------|----------|------|
| 创业/产品 | 解决真实用户的问题 | 涉及需求、市场、用户 |
| 技术/工程 | 技术方案选择、架构设计 | 涉及实现、性能、权衡 |
| 创意/探索 | 头脑风暴、新想法 | 没有明确约束 |

**创业/产品 → Phase 2A (深度提问)**
**技术/工程 → Phase 2B (方案发散)**
**创意/探索 → Phase 2C (自由发散)**

也可以混合: 比如一个 Android 功能需求，先用 Phase 2B 发散技术方案，再用 Phase 2A 的部分问题检验需求真实性。

### Phase 2A: 深度提问 (Forcing Questions)

针对创业/产品场景，逐个问以下问题。
**每次只问一个，等回答后再问下一个。**
**这不是可选的检查项，是核心流程。**

#### Q1: 需求真实性

"谁最需要这个东西? 不要说'用户'，说一个具体的人。他叫什么，做什么工作，什么时候会遇到这个问题?"

**追问:**
- 这个人现在怎么解决这个问题? (即使很笨)
- 他有没有为这个问题花过钱或时间?
- 如果这个东西消失了，他会怎样?

**红线:** 如果用户说"每个人都需要"或"市场很大"，这是没想清楚。继续追问。
**不跳过。** 这个问题是一切的基础。

#### Q2: 现状替代

"这个人现在用什么方法解决这个问题? 哪怕方法很烂。"

**追问:**
- 这个烂方法他用了多久了?
- 他有没有试过换别的方式?
- 为什么那些方式没成功?

**目的:** 理解竞争不是其他产品，而是用户当前的 workaround。
**不跳过。** 现状替代定义了你要超越的基准线。

#### Q3: 最小切入点

"如果只能做一个功能，就一个，用户会用什么理由为它买单?"

**追问:**
- 这个功能能独立存在吗? 还是必须依赖其他功能?
- 如果明天就要上线，你会保留什么?
- 什么功能砍掉之后，用户反而觉得体验更好了? (有时候少即是多)

**不跳过。** 最小切入点决定了你的第一版应该做什么。

#### Q4: 观察与意外

"你有没有实际观察过真实的人遇到这个问题的场景? 有没有什么让你意外的?"

**追问:**
- 用户有没有用过你没想到的方式?
- 有没有哪个行为模式让你重新理解了问题本身?

**Smart-skip:** 如果用户在 Phase 1 中已经描述了具体的用户观察，可以跳过。

#### Q5: 扩展空间

"如果这个做成了，下一步可以往哪个方向扩展? 哪个方向的回报最大?"

**追问:**
- 有没有什么东西，是你现在没做但以后可以自然延伸的?
- 什么变化发生的时候，这个东西会变得更有价值?

**Smart-skip:** 如果方向已经很明确且用户不想发散，可以跳过。

#### Q6: 未来适应性

"如果三年后世界变得不一样了，这个东西会变得更重要还是更边缘?"

**追问:**
- 什么趋势会让这个东西更被需要?
- 什么趋势会让它变得不重要?

**Smart-skip:** 如果用户明确表示是短期项目或内部工具，可以跳过。

**Smart-skip 规则总结:**
- Q1、Q2、Q3 **永远不跳过**。它们是发散思维的骨架。
- Q4-Q6 根据上下文决定是否跳过。
- 如果前面的问题自然覆盖了后面的问题，跳过。
- 如果用户表现出不耐烦 ("直接看方案吧")，从下一个不跳过的 Q 继续。
- 只有当用户提供了包含真实用户名、真实行为、真实数据的完整描述时，才能全部跳过。

### Phase 2B: 方案发散

针对技术/工程场景，发散不同的实现方案。

#### 步骤 1: 现有方案盘点

"现在有什么已有的方案可以参考? 包括开源项目、库、设计模式。"

扫描方式:
- 项目内已有的类似实现 (Phase 0 中已读取的代码结构)
- 用户提到过的技术偏好
- Phase 0 中检测到的技术栈

#### 步骤 2: 强制发散

生成至少 3 个不同方向:

**方向 A: 最小实现**
- 最少的代码、最少的依赖、最快的路径
- "如果只有半天时间，你怎么做?"

**方向 B: 理想方案**
- 不考虑时间成本，最好的实现是什么?
- "如果有无限时间，你会做成什么样?"

**方向 C: 侧向思考**
- 有没有完全不同的解决思路?
- "如果不能用常规方法，你会怎么做?"
- "反过来看这个问题，有没有被忽略的角度?"

对每个方向，简要说明:
```
方向 A: [名称]
- 思路: [一句话]
- 优点: [1-2 点]
- 代价: [1-2 点]

方向 B: [名称]
...

方向 C: [名称]
...
```

#### 步骤 3: 交叉碰撞

"如果把方向 A 的某个优点和方向 B 的某个优点结合起来，会得到什么?"

这步不是必须的，但经常能发现意外的好方案。

### Phase 2C: 自由发散

针对创意/探索场景，不设限制地探索。

#### 发散问题 (逐个问):

**"最酷的版本"**
"如果没有任何限制，这个东西最酷的版本是什么? 什么样的版本会让你自己想用?"

**"最意外的用途"**
"这个东西除了你预期的用途，还可能被用在什么地方? 有没有什么完全意想不到的使用场景?"

**"如果反过来"**
"如果把这个问题反过来想 (比如不是 A 导致 B，而是 B 导致 A)，会得到什么不一样的洞见?"

**"极端约束"**
"如果必须在一周内做完，你会怎么做? 如果必须在一行代码内解决呢?"

**"类比迁移"**
"其他领域有没有解决过类似的问题? 它们的方案能搬过来吗?"

**Smart-skip:** 用户给的初始描述如果已经覆盖了某个问题，跳过。

## Phase 3: 外部信息验证 (Landscape Awareness)

发散完成后，用外部信息检验"常规做法"是否正确。

### 步骤 1: 隐私门控

使用 AskUserQuestion:
> 我想搜索一下这个领域的外部信息，看看常规做法和最新趋势。这会发送泛化的搜索词 (不是你的具体想法)。可以吗?
> - A) 搜索
> - B) 跳过，只用已有知识

如果选 B，跳过整个 Phase 3。

### 步骤 2: 搜索

使用 WebSearch 搜索 (泛化关键词，不包含用户的特定想法):

**产品/创业场景:**
- "[问题领域] 常见做法 {当前年份}"
- "[问题领域] 常见错误"
- "why [现有方案] fails"

**技术/工程场景:**
- "[技术主题] best practices {当前年份}"
- "[技术主题] alternative approaches"
- "[技术主题] common pitfalls"

**创意/探索场景:**
- "[领域] existing solutions {当前年份}"
- "[领域] innovative approaches"
- "[领域] open source alternatives"

### 步骤 3: 三层综合

基于搜索结果，做三层分析:

**[Layer 1] 已知共识:**
> 大家都怎么做的? [1-2 句]

**[Layer 2] 最新趋势:**
> 搜索结果和当前讨论中有什么新变化? [1-2 句]

**[Layer 3] 逆共识检验:**
> 基于我们 Phase 1-2 的讨论，有没有常规做法是错的? [1-2 句]

**Eureka check:** 如果 Layer 3 发现了真正的反直觉洞察，明确标记:
> **EUREKA:** [洞察描述] → 意味着 [影响]

### 步骤 4: 重新评估

将外部信息反馈到之前的分析中:

使用 AskUserQuestion:
> 搜索结果有没有改变你对问题的看法?
> - A) 有，我想调整方向
> - B) 没有，常规做法验证了我们的方向
> - C) 有，但我想坚持我们的逆共识方向 (说明原因)

## Phase 4: 前提挑战

外部验证后，回过头来质疑所有前提。

### 逐项挑战

回顾 Phase 1-3 中的所有前提假设，逐个质疑:

**挑战方式:**
1. "我们一直在假设 [前提]。这个假设一定是真的吗?"
2. "如果 [前提] 是错的，整个方案会怎样?"
3. "有没有可能 [反面] 才是对的?"

**常见需要质疑的前提:**
- "用户需要 X" → 用户真的需要 X，还是他们想要 X 带来的结果?
- "技术上应该用 Y" → Y 是因为它是最好的，还是因为它最熟悉?
- "这个功能很重要" → 对谁重要? 权重排一下，它真的是 top 3 吗?
- "我们需要先做 Z" → Z 真的是前置条件，还是我们以为它是?
- "竞品没有这个功能" → 你确认过吗? 还是只是没看到?

**输出:**
```
前提质疑:
1. [前提] → 状态: 确认 / 被挑战 / 被外部信息推翻
2. [前提] → 状态: 确认 / 被挑战 / 被外部信息推翻
```

使用 AskUserQuestion:
> 这些前提挑战中，有没有让你重新思考的?
> - A) 有，让我调整方向
> - B) 没有，继续

## Phase 5: 独立视角 (Cross-Model Second Opinion)

通过 Claude subagent 获取一个独立的思考视角。

### 步骤 1: 组装上下文

将 Phase 1-4 的关键信息整理成结构化摘要:

```
主题: <用户最初描述>
目标: <Phase 1 的结论>
边界: <Phase 1 的边界>
核心前提: <Phase 4 后仍成立的前提>
发散方向: <Phase 2 的方案概要>
外部信息: <Phase 3 的关键发现>
关键用户回答: <用户说过的最有洞察的原话>
被推翻的前提: <Phase 4 中被外部信息推翻的前提>
```

### 步骤 2: 派发 subagent

使用 Agent 工具派发独立 subagent。subagent 有全新上下文，
看不到之前的对话，保证视角独立性。

**subagent prompt:**
```
你是一个独立的技术顾问。以下是一个头脑风暴会话的结构化摘要:

[上下文摘要]

你的任务:
1. 这个方向最大的盲点是什么? (所有人都在忽略的角度)
2. 有没有什么反直觉但可能是对的洞察?
3. 如果只有 48 小时和一个工程师，你会建议先做什么?
4. 有什么被忽略的风险?
5. [上下文中如果有 Eureka 洞察] 这个逆共识的洞察站得住脚吗?

直接回答，不要寒暄。每个回答 2-3 句话。
```

### 步骤 3: 交叉分析

将 subagent 的回答与主会话的分析做交叉对比:

```
独立视角 vs 主会话分析:
- 一致点: [双方都认为对的]
- 分歧点: [独立视角提出了不同的看法] → 原因: [为什么]
- 新洞察: [主会话中没出现过的角度]
- Eureka 验证: [如果有的话，独立视角怎么看]
```

如果独立视角挑战了某个前提:
> 独立视角挑战了前提: "[前提]". 理由: "[原因]"
> - A) 接受挑战，调整方向
> - B) 保持原方向

## Phase 6: 自我检验 (Adversarial Review)

在思考文档产出之前，用 subagent 做一次对抗性审查。

### 步骤 1: 派发审查 subagent

使用 Agent 工具派发第二个独立 subagent。这个 subagent 也看不到对话历史，
只能看到即将产出的思考文档。

**审查 subagent prompt:**
```
你是一个严格的评审者。阅读以下思考文档，在 5 个维度上挑刺:

[完整的思考文档内容]

对每个维度，标记 PASS 或列出具体问题:
1. **逻辑一致性** — 各部分之间有矛盾吗? 结论是否从前提中自然推导出来?
2. **隐含假设** — 有没有没被明说但被当作事实使用的前提?
3. **范围蔓延** — 是否超出了 Phase 1 确定的边界? 有没有混入"反边界"的内容?
4. **可操作性** — 这个思考文档能给后续执行提供清晰指引吗? 还是需要再想清楚?
5. **盲区检测** — 有没有整个讨论中都完全没涉及的方面?

最后给出质量分数 (1-10) 和最关键的 1-2 个改进建议。
```

### 步骤 2: 修复并重新审查

如果审查发现问题:
1. 修复思考文档中的问题
2. 重新派发审查 subagent
3. 最多 3 轮

**收敛保护:** 如果连续两轮审查返回相同的问题 (修复没解决或审查者不认同)，
停止循环，将未解决的问题标记为"待决事项"，保留在思考文档中。

如果 subagent 失败或超时，跳过自我检验，直接产出思考文档。
告诉用户: "自我检验不可用，直接产出思考文档。"

### 步骤 3: 审查结果报告

简短报告:
> 思考文档经过 N 轮自我检验。发现 M 个问题，已修复 M 个。
> 质量分数: X/10。
> [如果有未解决问题: 列出]

## Phase 7: 思考文档

将整个头脑风暴的结构化结果整理为思考文档。

### 产出选项

思考文档默认存在对话上下文中 (不写入文件)。在产出时询问:

> 思考文档已生成。如何保存?
> - A) 仅存内存 (下次 /clear 后丢失，适合快速探索)
> - B) 保存到文件 (可跨会话恢复)
> - C) 保存到文件并继续发散 (保存后回到某个 Phase 深入)

**如果选 A:** 直接在对话中输出思考文档。结束。

**如果选 B 或 C:** 写入文件，格式见下方。选 C 时跳到用户指定的阶段继续。

### 保存路径

```
docs/thinking/<日期>-<slug>.md
```

如果 `docs/thinking/` 不存在则创建。slug 从主题中提取 (取前 50 个字符，
转小写，空格转连字符，去掉特殊字符)。

### 恢复模式

使用 `/android-brainstorm resume <file>` 恢复已保存的思考文档:

1. 读取文件
2. 解析目标、边界、前提、方案方向
3. 检查 Phase 1-6 哪些已经完成
4. 从未完成的阶段继续
5. 如果 Phase 2 (发散) 已完成但用户想继续发散，可重新运行 Phase 2B/2C

### 后续衔接

思考文档产出后，根据用户意图建议下一步:

**如果目标是做功能/写代码:**
> "这个思考文档可以进一步细化为可执行 plan。建议:"
> - `/android-autoplan` (Android 项目: 拆分 + 审查 + 自动执行)
> - 手动编写 plan 后 `/android-worktree-runner import`

**如果目标还在探索:**
> "如果还需要继续发散，可以:"
> - `/android-brainstorm resume <文件>` 基于已有文档继续发散
> - 针对某个方案方向继续深入
> - 用不同的约束条件重新发散
> - 带着思考文档去找真实用户聊

**如果目标已经明确:**
> "方向已经清晰，下一步:"
> - 直接进入实现 (写 plan / 写代码)
> - 验证核心假设 (找用户聊 / 做原型)

## 异常情况处理

| 场景 | 处理方式 |
|------|----------|
| 用户目标太模糊 | 用具体例子帮用户聚焦，不替用户决定 |
| 用户跳过问题 | 尊重，直接跳到下一个阶段 |
| 发散方向太少 | 强制补充第三个方向 (侧向思考) |
| 所有前提都被推翻 | 回到 Phase 1 重新明确目标 |
| subagent 不可用 | 跳过对应 Phase，在思考文档中标注 |
| WebSearch 不可用 | 跳过 Phase 3，标注"未做外部验证" |
| 恢复文件不存在 | 提示用户检查路径，或新建主题 |
| 恢复文件内容损坏 | 读取可用部分，提示用户确认 |
